## 2. Методическая логика (кратко)

Тема учит видеть структуру: **два (или три) вида объектов → вклад каждого в общий признак → метод замены / перебора / диагностика**.

| Сложность | Что нового для ребёнка |
|-----------|------------------------|
| 1 | Базовая замена (головы, ноги, колёса, конечности) |
| 2 | Тот же паттерн на цену, количество, расход |
| 3 | Разная «производительность» участников; первые задачи без общего числа объектов |
| 4 | Положительные целые ограничения (`both_types_present`, запрет нуля) |
| 5 | Обязательная **письменная** запись решения (режим C+) |
| 6 | Три типа объектов или несколько допустимых ответов |
| 7 | Ключевое условие (равенство признаков), скрытая структура, неполнота данных |

**Жёсткие правила платформы:**

1. Ребёнок **не вводит** числа, которых нет в условии (кроме вычисленных им же промежуточных).
2. Если в тексте «мальчики и девочки», «собаки и кошки» и т.п. — `both_types_present: true`, `min_count: 1` для каждого типа.
3. Вариант с **0** одного из типов запрещён, если оба типа названы участниками.
4. Диагностика «данных не хватает» принимается только с **объяснением**, какого числа не хватает.
5. Числовой ответ **без** текстового решения не завершает задачу.
6. Для каждой задачи — **индивидуальный** массив `solution_lines`, не общий шаблон.

## 4. Модель данных задачи

Расширить `Task` (или вынести в `HeadsLegsTaskMeta`) полями из методички:

```typescript
interface HeadsLegsTaskMeta {
  /** Методический номер, напр. "1.1" */
  methodTaskId: string;
  difficultyLevel: 1 | 2 | 3 | 4 | 5 | 6 | 7;
  patternId: 1 | 2 | 3 | 4 | 5 | 6 | 7;

  entities: Array<{ id: string; label: string; emoji?: string }>;
  features: Array<{
    entityId: string;
    featureKey: string; // legs, wheels, price, ...
    value: number;
    /** Для производных признаков, напр. «правые ноги» = legs/2 */
    derive?: { from: string; formula: "half" | "custom"; expr?: string };
  }>;

  totals: {
    totalObjects?: number | null;      // null = нет в условии
    totalFeature?: number | null;
    extra?: Record<string, number>;    // глаза, руки, горбы...
  };

  constraints: {
    minCount?: 1;
    integerOnly?: true;
    bothTypesPresent?: boolean;
    allowZero?: boolean;
    /** Для 3+ типов */
    tripleType?: boolean;
  };

  /** A | B | C | D | E */
  solutionMode: "A" | "B" | "C" | "D" | "E";

  /** Уровень самостоятельности сборки модели (1–6 из методички) */
  modelIndependenceLevel: 1 | 2 | 3 | 4 | 5 | 6;

  solutionLines: SolutionLine[];
  acceptedAnswers: AcceptedAnswer;
  validationRules: ValidationRules;
  hintLevels: [string?, string?, string?];
}

type SolutionLine = {
  template: string;           // "Представим, что все [30] животных были [птицами]."
  blanks: SolutionBlank[];    // интерактивные [ ]
};

type SolutionBlank =
  | { type: "number"; accept: number | number[] }
  | { type: "object"; accept: string[] }
  | { type: "expression"; acceptPattern: string }  // regex / eval
  | { type: "conclusion"; keywords: string[] };

type AcceptedAnswer =
  | { kind: "single"; values: Record<string, number> }
  | { kind: "single_scalar"; value: number }
  | { kind: "multi_set"; sets: Array<Record<string, number>> }
  | { kind: "diagnostic"; requiredPhrases: string[] }
  | { kind: "text"; format: string };  // напр. "4:6" для МУХИ:СЛОНЫ

type ValidationRules = {
  numericTolerance?: 0;
  requiredSteps?: Array<
    "assumption" | "compute" | "compare" | "replacement_step" | "divide" | "answer" | "diagnosis"
  >;
  forbiddenZeroTypes?: string[];
};
```

**Маппинг на существующие поля `Task`:**

| Существующее | Новое использование |
|--------------|---------------------|
| `independenceLevel` | `modelIndependenceLevel` (пересмотреть шкалу 1–6) |
| `requiresUpload` | **`false` по умолчанию.** Если **`true`** — задача открывается в **режиме «только бумага»** (см. §10): цифровые шаги (`model_build`, `numeric_solve`, `word_solution`) **не показываются** |
| `skillWeights.modeling` | 0.7–1.0 для всех задач ветки |
| `stage` | = `difficultyLevel` |
| `maxStars` | 3 (без изменений) |

## 5. Новые типы шагов плеера

Текущие шаги (`drag_select`, `number_input`, `worksheet_table`, …) **не покрывают** методичку. Добавить:

### 5.1. `model_build` — сборка модели условия

| Уровень | UI |
|---------|-----|
| 1 | Карточки типов + drag чисел из условия в слоты модели; кнопка «данных не хватает» |
| 2 | Те же слоты, без подсказок-карточек названий |
| 3+ | Свободная таблица / поля; проверка ключевых чисел |

Слоты модели (настраиваемые per task):

- два вида объектов
- всего объектов (может быть пустым)
- общий признак
- признак вида 1 / вида 2

### 5.2. `numeric_solve` — числовой ответ

Единый шаг после модели:

- стандартная замена → одно или два числа;
- сравнение («на сколько больше») → число + опционально текст;
- диагностика → выбор «единственный ответ / данных не хватает / несколько ответов»;
- перебор (E) → таблица строк «если X = …, то …» с добавлением строк.

### 5.3. `word_solution` — «Запиши решение словами» (**обязательный**)

Подтипы по `solutionMode`:

| Режим | UI | Автопроверка |
|-------|-----|--------------|
| **A** | Drag карточек строк + карточек вставок в `[ ]` | Порядок строк + значения blanks |
| **B** | Фиксированные строки, заполнение `[ ]` | Значения blanks |
| **C** | Свободный ввод построчно или одно поле | `requiredSteps` + ключевые числа из `solutionLines` |
| **D** | Вывод + объяснение; чек «какого числа не хватает» | Keywords + запрет zero-вариантов |
| **E** | Таблица перебора + итоговая строка | Минимум 2 проверенных строки + корректный вывод |

**Общее:** финальная строка «Ответ: …» или «Вывод: …» обязательна.

### 5.4. Алгоритм автопроверки текста (режим C/D/E)

1. Извлечь числа из ввода ребёнка.
2. Сравнить с этalon-числами из `solutionLines` (допуск: перестановка строк, синонимы объектов).
3. Проверить наличие `requiredSteps` (regex-наборы: «представим», «по условию», «значит», «ответ»).
4. Для D: обязательны фразы недостаточности + **отсутствие** выдуманного «всего N объектов».
5. Для E: минимум 2 строки перебора с корректной арифметикой OR совпадение с одной строкой этalona.
6. При частичном совпадении — подсказка уровня 2, не сразу fail.

## 6. Сценарий прохождения одной задачи

Два **взаимоисключающих** сценария в зависимости от `requiresUpload`:

### 6.1. Цифровой режим (`requiresUpload: false`) — по умолчанию

```
read_condition
    ↓
[optional] condition_parse — только если enableGivenStep
    ↓
model_build
    ↓
numeric_solve  ←── может быть пропущен для чистой D (7.6)
    ↓
word_solution  ←── ВСЕГДА, solutionMode A–E
    ↓
[optional] auto_explanation
    ↓
завершение, начисление звёзд (автопроверка)
```

### 6.2. Бумажный режим (`requiresUpload: true`)

```
read_condition  ←── условие + инструкция «письменно, развёрнуто»
    ↓
[optional] «📷 Загрузить фото» — на том же экране (до или после «Готово»)
    ↓
«Готово, решил на бумаге»  →  статус «На проверке» (НЕ «Выполнено»)
    ↓
[optional] экран/блок «Загрузить фото решения» — если не загрузили раньше
    ↓
проверка методистом / родителем  →  «Выполнено» или «Переделать»
```

**В бумажном режиме запрещено:** `model_build`, `numeric_solve`, `word_solution`, подсказки к решению, progress «шаг N из M» по цифровым шагам, **автоматическое** проставление «Выполнено» и звёзд.

**Звёзды — цифровой режим:**

| Звёзды | Критерий |
|--------|----------|
| 3★ | Модель + число + текст с 1-й попытки |
| 2★ | С подсказками / 2-я попытка |
| 1★ | Завершено с помощью |

**Звёзды — бумажный режим:**

| Этап | Звёзды / прогресс |
|------|-------------------|
| После «Готово, решил на бумаге» | **Не начисляются**; в карте задачи — **«На проверке»** |
| После подтверждения проверяющим | **1–3★** по качеству решения (критерии задаёт методист при проверке) |
| «Переделать» | Сброс попытки; ребёнок снова видит условие и может отправить заново |

> Аналогия с Daily: `pending` → `approved` / `redo` (см. §10.6).

## 10. Бумажный режим (`requiresUpload` / `paper_only`)

**Решение (v1.3):** галочка **`requiresUpload`** переключает задачу в режим «только бумага». Цифровые шаги **не показываются**. **Выполнено** и звёзды — **только после проверки**.

### 10.1. Когда включается

| Аспект | Правило |
|--------|---------|
| По умолчанию | `requiresUpload: false` — полный цифровой сценарий §6.1 |
| Включение | Галочка **«Решение на листочке / в тетради»** в админ-панели |
| Назначение | Методист **выборочно** назначает задачи для проверки **письменного** мышления на бумаге |

### 10.2. Экран задачи

**До отправки** на экране:

1. **Карточка условия** — номер, название, полный текст (`condition`).
2. **Информационный блок** (шаблон + опциональный `paperPrompt`):

   > Реши задачу **письменно** на листочке или в тетради.  
   > Запиши **развёрнутое решение**: ход рассуждения, вычисления и ответ полными предложениями.  
   > Числовой ответ без объяснения не засчитывается.

3. **Не показывать:** цифровые шаги решения, подсказки к решению, progress-bar по шагам.

4. **Действия ребёнка** (без отдельного «окна решений»):

   | Элемент | Обязательность |
   |---------|----------------|
   | **«Готово, решил на бумаге»** | Обязательно — отправка на проверку |
   | **«📷 Загрузить фото решения»** | **Опционально** — два допустимых места (см. ниже) |

### 10.2.1. Загрузка фото (опционально)

Фото **не обязательно**, но если ребёнок/родитель хочет приложить работу:

| Вариант | Когда |
|---------|--------|
| **A** | На **том же экране**, что условие — блок загрузки виден сразу (рядом с инструкцией) |
| **B** | **После** нажатия «Готово, решил на бумаге» — переход на экран/блок «Загрузить фото решения» с кнопками «Загрузить» и «Пропустить» |

Оба варианта допустимы в одном UI: до «Готово» можно загрузить заранее; после «Готово» — предложить загрузить, если фото ещё нет.

**Важно:** нажатие «Готово» **не означает** выполнение задачи — только **сдачу на проверку**.

### 10.3. Поведение плеера (`TaskPlayer`)

```typescript
if (task.requiresUpload) {
  // Цифровые task.steps игнорируются
  // UI: PaperTaskScreen — condition + instruction + optional upload + «Готово»
  // После «Готово» → paperReviewStatus: "pending", НЕ completedTasks
}
```

- Бейдж: **«📝 письменно»**; после сдачи — **«На проверке»**.
- Сессия: `mode: "paper"`, `paperReviewStatus`, optional `paperUpload`.
- **`completedTasks` / звёзды / прогресс ветки не обновляются** при «Готово».
- `auto_explanation` — только после `approved` (**F5+**).

### 10.6. Статусы проверки (обязательно)

По аналогии с Daily (`daily-subject-status.ts`):

| Статус UI | Код | Когда |
|-----------|-----|--------|
| *(пусто)* | `not_started` | Задача не открывалась / не сдана |
| **На проверке** | `pending` | После «Готово, решил на бумаге» (+ опционально фото) |
| **Выполнено** | `approved` | Методист / родитель **подтвердил** решение |
| **Переделать** | `redo` | Проверяющий отклонил; ребёнок проходит заново |

**Правила:**

- До `approved` задача **не считается выполненной** в карте ветки, ачивках и отчёте родителя.
- Кнопки проверки: в **Telegram** (как daily) и/или в **админке / родительском экране** (**F5**).
- При `redo`: очистить `pending`-сессию, показать условие снова; старое фото можно заменить.

```typescript
type PaperReviewStatus = "not_started" | "pending" | "approved" | "redo";
```

### 10.4. Админ-панель

| Поле | Описание |
|------|----------|
| `requiresUpload` | boolean — включает бумажный режим |
| `paperPrompt` | optional string — дополнение к стандартной инструкции §10.2 |

При `requiresUpload: true` редактор **цифровых шагов** в админке может быть скрыт или помечен «не используется в бумажном режиме».

### 10.5. Отличие от цифрового режима

| | Цифровой | Бумажный |
|--|----------|----------|
| UI решения | Есть (A–E) | **Нет** |
| Автопроверка | Да | **Нет** |
| Фото | — | Опционально (до или после «Готово») |
| После действия ребёнка | Сразу «Выполнено» + звёзды | **«На проверке»** |
| Финальный статус | Авто | Только после **подтверждения** проверяющим |

## 11. Критерии готовности релиза

1. В ветке ровно **51** задача, порядок по методическим номерам.
2. В **цифровом** режиме ни одна задача не завершается без шага `word_solution`.
3. В **бумажном** режиме — только условие §10.2; **«Выполнено»** не ставится до `approved` §10.6.
4. Бумажный режим: фото опционально (§10.2.1); после «Готово» — **«На проверке»**.
5. Режимы A–E реализованы и покрыты минимум по 2 задачи каждый (цифровой режим).
6. Автопроверка текста работает без участия родителя для цифрового режима.
7. Особые задачи §3.1 проходят QA.
8. `taskCount` и прогресс ветки соответствуют 51 (**прогресс = только `approved` задачи**).
9. Smoke-тест цифрового и бумажного режима (в т.ч. цикл pending → approved / redo).
