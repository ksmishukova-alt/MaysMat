1. Объём проверки и источники
Проверялась актуальная локальная копия проекта из архива MaysMat-main (3).zip и live-страница production-домена. Production-главная открывается и показывает карту тем, включая «Головы и ноги», «Дирихле», «Метод неудачника» и другие ветки. Кодовая проверка проводилась по текущему проекту: task data, route guards, branch filters, QA scripts, TaskPlayer и PaperTaskScreen.
Всего в текущем TASKS: 272 задачи.
Детский видимый маршрут: 40 задач.
Олимпиадный архив: 63 задачи.
Blocked: 4 задачи, все связаны с missing_visual_asset / manual_review.
needsReview: 87 задач.
QA-скрипты проходят: qa:task-quality, qa:dirichlet, qa:categories, qa:task-access, qa:smoke.
2. Executive summary
Продукт сделал важный шаг: publishTier/qaStatus уже реально фильтруют детские списки задач, а прямой детский доступ к archive/hidden/methodistOnly закрыт экраном «Задача недоступна».
Но текущая версия ещё не соответствует заявленной образовательной методике полного курса: нет видео перед большинством тем, нет бумажных задач 16-20, нет PDF-бланков, нет загрузки решения с подтверждением ParentID.
В детский маршрут попадает 40 задач, но постепенное снятие опоры реализовано слабее, чем задумано: большинство видимых задач всё ещё имеет 8-10 экранов и повторяет одинаковый guided-flow.
Скрытый/архивный банк нельзя считать методически готовым: 63 archive задачи имеют needsReview, а многие используют generic «зайцы/клетки» flow даже там, где нужен отдельный flow для конструкций/контрпримеров.
Самый опасный технический риск: methodist mode включается query-параметром ?mode=methodist и без авторизации открывает все задачи. Для production это нужно закрыть feature flag или ролью.

6. Проверка в роли ребёнка 10 лет
Что хорошо: задача открывается с чтения условия, есть видимые шаги, есть подсказки, карточки и проверяемые поля, финальный эталон.
Где ребёнок застрянет: в задачах на доказательство язык слишком быстро становится взрослым: N/M, «клетки», «переполнение», «обобщённый принцип» без постоянной детской переформулировки.
На task 1.1 Dirichlet логика понятна, но фраза «Клетки — группы по признаку из условия» в proof lines слишком обобщённая; ребёнку лучше видеть «возрасты 20, 21, ..., 35».
На heads-legs ребёнок долго решает почти одинаковые 9-экранные задачи. Это тренирует повторение, но быстрее утомляет, чем методическая задумка с постепенным сокращением окон.
Ветка «Конструкции» по умолчанию пустая и показывает архивный toggle. Для ребёнка это лучше скрыть из карты тем или показать как «скоро/для старших», иначе непонятно, почему тема есть, а задач нет.
Нет очевидного механизма «реши на бумаге»: ребёнку не предлагают скачать бланк, распечатать, загрузить решение и ждать ParentID/родительскую проверку.
7. Проверка в роли senior методолога
Методическая траектория стала лучше: T1 начинается с простых OK-задач, T2 вынесен в «Метод неудачника», archive отделён.
Но структура «1 задача объясняет всё → 2-3 сокращают → 4-5 спрашивают следующее действие → 12-15 вписывают выражения → 16-20 решают на бумаге» пока не реализована как data-driven профиль.
В child-route нет ни одной задачи paper-mode, поэтому последняя стадия самостоятельности отсутствует.
Видимых задач по Дирихле всего 8 + 1 в unlucky; это скорее demo-траектория, а не полноценная тема.
Скрытые/архивные задачи нельзя проверять как готовую методологию: у них правильно выставлен статус archive/needsReview, но многие уже имеют сгенерированные шаги. Методисту нужно явно видеть, что это черновой flow.
Рекомендуется ввести themePathVersion: demo / course / archive. Сейчас ветки выглядят как полноценные темы, даже если внутри 0-8 готовых задач.

9. UX/UI senior audit
Сильные стороны: визуально дружелюбная оболочка, понятные карточки, мягкая палитра, есть навигация по теме, статус недоступной задачи.
Проблема 1: недостаточно видно метод. Нужно добавить постоянную ленту метода: «кто участвует → куда раскладываем → считаем → сравниваем → вывод → запись».
Проблема 2: названия задач обрезаются в child list и h2. Нужен childTitle + fullTitle/condition отдельно.
Проблема 3: в ветках с 0 детских задач лучше показывать не пустой список, а экран «Тема готовится» или скрывать такие ветки с детской карты.
Проблема 4: видео — заглушка и только у двух веток. Для UX это обещание, которое не выполняется в большинстве тем.
Проблема 5: paper upload screen существует, но не встроен в маршруты; кнопки «Скачать бланк» нет.
10. Архитектурный аудит
Хорошо: есть единая Task model, step discriminated union, publishTier/qaStatus, branch filters, TaskUnavailableScreen, route smoke tests.
Риск: data хранится в TypeScript и manual overrides. Для методиста без разработчика это трудно поддерживать. Нужен нормализованный JSON/CMS layer или admin editor.
Критично: ?mode=methodist даёт доступ ко всем задачам без auth. В production это должно быть закрыто.
Риск: QA основан на regex-сканировании текста. Это полезно, но не гарантирует методическую корректность flow.
Риск: visual asset pipeline отсутствует. Нет схемы: task.requiresVisual -> assetId -> alt -> render -> fallback.
Риск: бумажный режим не имеет генератора PDF-бланка и идентификатора родительской проверки.
11. Системно-аналитический аудит
Нужно формально описать state machine: draft -> qaReview -> ready -> childRoute/training -> archived/hidden.
Нужно разделить роли: child, parent, methodist, admin. Сейчас methodist mode - query parameter.
Нужно описать event model: video_seen, task_started, hint_used, step_failed, paper_blank_downloaded, paper_solution_uploaded, parent_review_requested, parent_approved/redo.
Нужно связать ParentID с загрузкой решения и статусом проверки. Сейчас user-profile имеет parentId=demo-parent, но PaperTaskScreen не показывает/не подтверждает это ребёнку.
Нужно добавить acceptance criteria для каждого publishTier: что видно в списке, что видно по прямой ссылке, что доступно с archive toggle, что доступно только в admin.

13. Приоритетный backlog
P0 · P0-01 · methodist mode доступен query-параметром без авторизации
Доказательство: canAccessTask(mode="methodist") возвращает true для любой задачи; parseTaskAccessMode читает mode из URL. При отключенной авторизации любой пользователь, знающий URL, может открыть ?mode=methodist.
Что сделать: В production отключить methodist mode через feature flag или требовать реальную роль/секрет. Для ребёнка прямые ссылки на hidden/blocked/archive уже закрыты, но methodist bypass надо убрать из публичного контура.
P0 · P0-02 · нет реализации бумажных задач 16-20: бланк PDF + загрузка + parentId
Доказательство: В TASKS: requiresUpload=true найдено 0 задач; hasDownload=true найдено 0 задач. PaperTaskScreen умеет загрузку фото, но нет PDF-бланка, нет привязки к ParentID в отправке.
Что сделать: Добавить тип этапа paper_solution для задач 16-20 внутри подтем: downloadBlankUrl, uploadRequired/optional, parentReviewId, status pending/approved/redo. Генерировать PDF-бланк с условием и местом под решение.
P0 · P0-03 · нет видеообъяснений перед большинством тем/подтем
Доказательство: BRANCH_VIDEOS содержит только modeling-heads-legs и fairy-caves. У dirichlet/unlucky/constructions/remainders/cutting/tables/relations/coloring/extreme нет intro video, хотя методика требует видео перед каждой темой/подтемой.
Что сделать: Сделать обязательный videoMeta для каждой публикуемой темы и подкатегории: intro, methodDemo, beforeSubtopic. До готовности видео показывать scripted explanation screen, но не молча пропускать этап.
P0 · P0-04 · архивные/скрытые задачи не методически готовы, но доступны в preview/methodist
Доказательство: 63 archive задачи имеют qaStatus=needsReview и too_hard_for_child_route; многие используют generic flow с «зайцы/клетки» даже для конструкций/контрпримеров. Для ребёнка скрыто, но методистский preview может создавать ложное впечатление готовности.
Что сделать: В archive preview явно ставить watermark «черновик/методистский банк». Не использовать детский task-runner для F10_ADVANCED, пока не разработаны отдельные flows для конструкций, графов, крайних задач.
P1 · P1-01 · снятие опоры почти не реализовано внутри видимого маршрута
Доказательство: В детском маршруте 40 задач; 26 из них имеют 9 шагов, у heads-legs 23 из 28 задач имеют 9 шагов. На 16-20 задачах нет перехода к бумажному полному решению.
Что сделать: Ввести progressionProfile по каждой подтеме: 1 задача - объяснение; 2-3 - тренировка с меньшим числом окон; 4-5 - выбор следующего действия; 12-15 - вписать выражения; 16-20 - бумажное решение.

15. Вердикт
Текущий сайт можно использовать как внутренний прототип и как демо первой волны задач. Для детского тестирования в ограниченном режиме можно давать только видимые childRoute задачи, но при условии закрытия methodist mode и явного предупреждения, что видео/бумажный режим ещё не реализованы. Для полноценной образовательной платформы нужно сначала довести контур методической самостоятельности: видео -> guided explanation -> сокращение шагов -> выбор следующего действия -> вписывание выражений -> бумажное решение с проверкой родителя.
